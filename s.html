The actual diagnosis: You've been learning patterns by example. You can recognize the exact problems you've seen. But recognition by memory is not the same as detection by logic. You need to switch from "does this look like problem X" to "what properties does this problem have, and which pattern owns those properties."

Why repeated practice on same problems doesn't transfer

1
You see Subarray Sum = K → you remember prefix sum.
But you learned "Subarray Sum = K uses prefix sum" not "problems that ask for count of subarrays with a property, use prefix sum + hashmap." The rule is too narrow.
2
LeetCode throws "Count subarrays with product less than K."
Same underlying idea (sliding window or prefix product), different surface. You freeze because it doesn't look like the problem you memorized.
3
The fix is not more problems. The fix is abstraction.
You need to learn the abstract "shape" of each pattern — what structural properties does the problem have when a pattern applies? That shape detection works on any new problem.

The two kinds of pattern learners

Surface learner (you right now)Deep learner (target)
"Subarray Sum K → prefix sum""Count of pairs with sum = target in a continuous sequence → prefix sum + hashmap, because I need to know how many prefixes I've seen before"
"Course schedule → topo sort""Any problem with dependencies where I need to find order, or detect circular dependency → model as directed graph, topo sort if DAG"
"Longest common subsequence → 2D DP""Any problem comparing two sequences where I need to find optimal matching/alignment → 2D DP because state = (position in seq1, position in seq2)"
"Koko eating bananas → binary search on answer""Any problem where answer is a monotonic value in some range, and I can check feasibility of a candidate answer in O(n) → binary search on the answer space"
The deep learner sees a new problem and runs structural analysis. The surface learner tries to find a matching problem they've seen before.
Run these 5 questions on every new problem. In order. Every time. This is your detection protocol. After 30 problems with this protocol, it becomes reflex.
1
Q1: What is the input shape?
Array (sorted? unsorted?) → Binary search, two pointer, sliding window candidates
String → Sliding window, hashing, DP on strings, Trie
Graph / adjacency list → BFS, DFS, Dijkstra, Union-Find
Tree → Recursion, DFS, BFS level-order
Two sequences → 2D DP (LCS family)
Grid (matrix) → BFS flood fill, DFS, DP on grid
Set of items with weights/values → Knapsack DP
2
Q2: What is the question actually asking?
Count something → Prefix sum, hashmap, DP counting
Find min/max → DP optimization, greedy, Dijkstra
Find all (enumerate) → Backtracking
Is it possible / valid? → BFS/DFS reachability, Union-Find connectivity
Find order / sequence → Topo sort
Find kth element → Heap, QuickSelect
Detect cycle → Floyd's, DFS with color, Union-Find
3
Q3: Are there constraints I should exploit?
"Subarray" or "contiguous" → Sliding window or prefix sum (not backtracking)
"Subsequence" (not contiguous) → DP or two pointer
"Sorted input" → Binary search, two pointer
"No extra space" → In-place techniques, Floyd's
"Distinct / unique" → Hashset
"At most K" or "exactly K" → Sliding window with constraint
"All permutations / combinations" → Backtracking
"Minimum cost to connect" → MST (Kruskal/Prim)
"Shortest path with weights" → Dijkstra / Bellman-Ford
4
Q4: Can I define "optimal substructure"?
Ask yourself: "If I knew the answer for a smaller version of this problem, could I build the answer for the full problem?"

YES → DP or recursion + memo. Now define: what is the smallest subproblem? That is your dp[i] state.
NO → Greedy or direct simulation is likely enough.

This single question is the entire secret to knowing when to use DP.
5
Q5: What is the brute force, and what is wasteful in it?
Always write brute force first mentally.
Then ask: "What am I recomputing over and over?"

If you recompute overlapping subproblems → Memoize → DP
If you recompute prefix sums → Store in hashmap
If you linearly scan for next greater/smaller → Use a monotonic stack
If you repeatedly find min/max → Use a heap
If you search in a sorted space → Binary search

The optimization is always about eliminating the redundant work in brute force.

Apply this to a real example — "minimum path sum in a grid"

Q1: Input shape → 2D grid (matrix). Candidates: grid DP, BFS.
Q2: Question → find minimum (optimization). Greedy or DP.
Q3: Constraints → "top-left to bottom-right, only right/down moves" — no branching choices, strict direction.
Q4: Optimal substructure → min path to (i,j) = min(path to (i-1,j), path to (i,j-1)) + grid[i][j]. YES → DP.
Q5: Brute force recomputes same cells → memoize each cell's min path.

Result: 2D DP, dp[i][j] = grid[i][j] + min(dp[i-1][j], dp[i][j-1]).
You didn't need to remember this problem. You derived the pattern from the structure.
These are the structural signals — properties of the problem, not keywords in the title. Train yourself to notice these in the first 2 minutes of reading any problem.

Signal → Pattern mapping

Signal: "contiguous subarray" + "find count/length/sum"
Sliding Window when you need max/min length or a condition over a window
Prefix Sum + HashMap when you need to count subarrays with exact target sum
Key difference: "at most K" → sliding window. "exactly K" → prefix sum.
Signal: sorted input OR problem asks to "minimize the maximum" or "maximize the minimum"
Binary Search — specifically binary search on the answer space.
The trick: if you can check "is X a feasible answer?" in O(n), and the answer space is monotonic, binary search on it.
Examples: Koko bananas, split array largest sum, capacity to ship packages — all same skeleton.
Signal: "next greater", "previous smaller", "span", "histogram"
Monotonic Stack
Maintain a stack that is always sorted (increasing or decreasing). When you break the order, pop and process.
If it's about a range/window maximum → Monotonic Deque
Signal: "k-th largest/smallest", "top K", "median", "streaming data"
Heap (PriorityQueue)
Two heaps for median. Min-heap for top K largest. Max-heap for K smallest.
If it's "merge K sorted" → heap with (value, list_index, element_index).
Signal: "all subsets", "all permutations", "all paths", "generate all valid combinations"
Backtracking
Template: choose → recurse → unchoose. Add pruning when constraint is violated.
Key: if problem asks for COUNT of valid combinations, switch to DP (backtracking is exponential).
Signal: "overlapping intervals", "meeting rooms", "events with start/end"
Intervals — sort by start
Merge: iterate and compare end of last vs start of current.
Count max overlap: sweep line with events array or min-heap of end times.
Insert: binary search for position, then merge.
Signal: "find duplicate", "detect cycle in linked list", "missing number"
Floyd's / XOR / Index-as-hashmap
No extra space → Floyd's tortoise and hare for cycle.
Missing/duplicate in range [1..n] → index marking or XOR trick.

The "at most K" vs "exactly K" signal — most misused pattern

"at most K distinct chars" → shrinkable sliding window (expand right, shrink left when violated)
"exactly K distinct chars" → f(exactly K) = f(at most K) - f(at most K-1) [TWO window calls!]
"at most K replacements" → sliding window (LC 424 longest repeating char replacement)
"exactly K operations allowed" → usually DP with state (index, k_remaining)
This single distinction explains why you solve "at most K" correctly but freeze on "exactly K".
DP signal detection is all about identifying what changes between subproblems. That "thing that changes" is your state. Once you have the state, the recurrence almost writes itself.
Signal: "how many ways to reach / make / partition"
Counting DP — dp[i] = number of ways
Coin change II, unique paths, target sum, stair climbing.
Recurrence: dp[i] += dp[i - something] for each valid choice.
Signal: "minimum/maximum cost / length / deletions to achieve X"
Optimization DP — dp[i] = min or max value
Edit distance, coin change (min coins), longest increasing subsequence (LIS → max length).
Recurrence: dp[i] = min/max(dp[j] + cost) for valid j.
Signal: TWO sequences being compared / aligned / matched
2D DP — LCS family
State: dp[i][j] = answer considering first i chars of s1 and first j chars of s2.
LCS, Edit distance, Wildcard matching, Distinct subsequences, Interleaving strings.
Base cases: dp[0][j] = j (deletions), dp[i][0] = i (insertions).
Signal: "partition array into two equal halves" or "can we pick items summing to X"
0/1 Knapsack
Each item can be taken or left. dp[j] = can we achieve sum j?
Partition equal subset (LC 416), Last stone weight II (LC 1049), Target sum.
Run backwards when using 1D array to avoid reusing same item.
Signal: "interval" — problem defined on subarray [i..j] where answer for [i..j] depends on answers for smaller intervals
Interval DP
State: dp[i][j] = answer for subarray from index i to j.
Fill diagonally: length 1 first, then 2, then 3... up to full length.
Burst balloons, minimum cost to cut stick, palindrome partitioning.
Signal: "at each position, decide include/skip with some restriction on adjacent"
Decision DP — House Robber family
dp[i] = best you can do considering first i elements.
dp[i] = max(dp[i-1], dp[i-2] + nums[i]).
Circular → run twice: [0..n-2] and [1..n-1], take max. On tree → DFS returning (take, skip).
Signal: "longest increasing subsequence" style — find longest chain where each element satisfies a condition with the previous
LIS pattern
O(n²) DP or O(n log n) with patience sorting / binary search.
Applies to: LIS (LC 300), Russian doll envelopes (LC 354), longest chain of pairs.
Key: sort by one dimension, LIS on the other.

How to define dp state when stuck

Step 1: Ask "what would I need to know at position i to make the optimal decision?"
Step 2: Everything you "need to know" becomes dimensions of your state.
Step 3: If state has 1 thing → 1D dp[i]. If 2 things → 2D dp[i][j].

Example: "Longest path in grid where each step must be strictly increasing"
At cell (i,j): I need to know → which cell I'm at. That's it → dp[i][j] = longest path starting from (i,j).

Example: "Min cost with at most K transactions (stocks)"
At day i with k transactions left: I need → (day, k, holding or not) → dp[i][k][0/1].
Graph pattern detection starts with one question: "What is a node, and what is an edge?" Many problems are secretly graph problems once you model them correctly. If you can't answer this, you haven't modeled the problem yet.
Signal: "find all connected components" or "how many groups" or "are A and B in the same group"
DFS / BFS component counting OR Union-Find (DSU)
Use DSU when: unions happen online (one at a time), you need fast "same group?" queries.
Use BFS/DFS when: you need to process/label each component's members.
Examples: Number of islands, Number of provinces, Accounts merge.
Signal: "shortest path" — unweighted graph
BFS — always BFS for unweighted shortest path
BFS guarantees shortest path in unweighted graphs because each level = one more edge.
Multi-source BFS: put ALL sources in queue at start, they expand simultaneously.
Examples: 01 Matrix, Rotting oranges, Word ladder.
Signal: "shortest path" — weighted graph with non-negative weights
Dijkstra — min-heap + visited set.
State in heap: (cost, node). Process cheapest first.
Variant: if edge weights are 0 or 1 only → use deque (0-1 BFS), O(V+E) instead of O(E log V).
Examples: Network delay time, Path with minimum effort, Cheapest flights (with K stop constraint → Bellman-Ford).
Signal: "dependencies" / "prerequisites" / "ordering" / "can we finish all tasks"
Topological Sort (Kahn's BFS or DFS)
Model: tasks = nodes, dependency A→B = directed edge.
If cycle exists → impossible (return empty).
Kahn's: start from nodes with in-degree 0, process one layer at a time.
Examples: Course schedule, Alien dictionary, Sequence reconstruction.
Signal: "minimum cost to connect all nodes" or "minimum spanning tree"
Kruskal (sort edges + DSU) or Prim (heap-based)
Kruskal: sort all edges by weight, add edge if it connects two different components.
Examples: Min cost to connect all points (LC 1584), Optimize water distribution.
Not Dijkstra — Dijkstra finds shortest path from source, MST connects all nodes.
Signal: "can we color graph with 2 colors" or "split into two groups with no internal conflict"
Bipartite check — BFS/DFS coloring
Try to 2-color with BFS. If neighbor has same color → not bipartite.
Examples: Is graph bipartite (LC 785), Possible bipartition (LC 886).
Equivalent: can you separate nodes into two sets where all edges cross between sets?
Signal: "the problem is NOT a graph but can be MODELED as one"
This is the hardest detection. Non-obvious graph problems:
Word ladder: words = nodes, 1-char-different = edge → BFS
Open the lock: each lock state = node, one turn = edge → BFS
Escape the maze: position = node, each move = edge → BFS
Gene mutation: gene string = node, one mutation = edge → BFS
Pattern: if the "state" changes step by step and you want min steps → model as BFS on state space.

DFS color states — critical for cycle detection

WHITE (0) = not visited
GRAY (1) = currently in recursion stack (being processed)
BLACK (2) = fully processed

Cycle in DIRECTED graph: if you visit a GRAY node → cycle exists.
(Visiting a BLACK node is fine — it's already been processed.)

Cycle in UNDIRECTED graph: if you visit any visited node that is NOT your parent → cycle.
Most people only know "visited = true/false" and get wrong answers on directed graphs. GRAY vs BLACK is the fix.
Trees are recursion problems. Every tree problem has one recursive structure: solve for left subtree → solve for right subtree → combine at root. The only question is what you return from each call.
Signal: "find a value / path / node at a specific depth" or "level-by-level processing"
BFS (level-order) — use queue, process layer by layer.
Right side view → take last element of each level.
Level averages → sum / count per level.
Connect next right pointers → process pairs within each level.
Signal: "path from root to leaf" or "path sum" or "root-to-leaf string"
DFS with running state passed down
Pass current path / current sum as parameter going DOWN the tree.
At leaf: check if condition is met.
Examples: Path sum II, root-to-leaf numbers, binary watch.
Signal: "any path" (not necessarily root-to-leaf) or "diameter" or "maximum path sum"
DFS returning value UP + tracking global answer
At each node: compute best you can extend upward (return to parent), but also check if the node can be the "peak" of a path going left-through-root-right.
This "left + root + right" combination is the answer candidate — update global max.
Return only the single-branch extension to parent.
Examples: Diameter (LC 543), Max path sum (LC 124), Longest zigzag.
Signal: "Lowest Common Ancestor" of two nodes
Post-order DFS
Key insight: LCA is the first node where p and q are in different subtrees (or one of them IS the node).
If left subtree found one and right subtree found the other → current node is LCA.
If both found in one subtree → LCA is deeper (return that result up).
For BST: if both < root → go left. If both > root → go right. Otherwise root is LCA.
Signal: "construct tree from traversal" or "serialize/deserialize"
Preorder + Inorder → construct
First element of preorder = root. Find root in inorder → left/right split.
Key: use hashmap for O(1) index lookup in inorder array.
Serialize with preorder (or level-order with nulls). Deserialize: reconstruct same way.
Signal: BST problem — "kth smallest/largest", "range sum", "validate", "insert/delete"
Inorder traversal gives sorted order
Kth smallest: inorder, count till k (or Morris traversal for O(1) space).
Validate BST: inorder must be strictly increasing, OR pass (min, max) bounds top-down.
Range sum: if root < low → go right only. If root > high → go left only. Both sides otherwise.

The one template that covers 80% of tree problems

int solve(TreeNode node) {
if (node == null) return BASE_CASE; // define: 0, null, -inf, +inf, false

int left = solve(node.left); // get answer from left subtree
int right = solve(node.right); // get answer from right subtree

// Update global answer if path THROUGH this node is a candidate
globalAns = max(globalAns, left + node.val + right); // or whatever combines

// Return: what can I offer to MY parent? (usually single branch)
return node.val + max(left, right);
}
Diameter, max path sum, longest zigzag, count good nodes — all use this exact skeleton. What changes is BASE_CASE, the globalAns update formula, and the return expression.
The transfer drill. This is the training method that actually builds pattern detection — not more problem solving, but structured analysis of problems you haven't solved yet.

The 10-minute analysis drill (do this on 2 new problems daily)

1
Read the problem statement. Set a 2-minute timer. Do NOT code.
In these 2 minutes: write down your answers to Q1 and Q2 from the framework (input shape, what is asked). Just those two.
2
Write one sentence: "This problem is asking me to _____ on a _____ structure."
Example: "This problem is asking me to COUNT subarrays on an array structure." This forces you to abstract before you pattern-match.
3
Now apply Q3 (constraints), Q4 (optimal substructure), Q5 (what's wasteful in brute force).
After 5 minutes of analysis, write your predicted pattern. Don't solve yet.
4
Check the editorial or tag — were you right?
If YES: implement it. The win is the analysis, not the code.
If NO: figure out WHICH question you answered wrong in the framework. That question is your weak point — not the pattern.
5
Write in your journal: "Signal I missed → pattern it maps to."
After 30 of these drills you have 30 signal-to-pattern mappings in your own words. That is your personal signal dictionary. Review before every mock.

The "near-miss" analysis — most important habit

When you solve a problem wrong or freeze: don't just read the solution. Do this instead:
1. Which of the 5 questions did you NOT ask?
2. Which constraint did you overlook?
3. What was the signal in the problem that you should have detected?
4. Write: "Next time I see [signal], I will think [pattern] because [reason]."
This is the only form of deliberate practice that improves detection speed. Reading solutions improves implementation. Near-miss analysis improves detection.
